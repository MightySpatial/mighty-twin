MightyTwin · Desktop & Tablet Coherence

Brings the mobile pattern set (see mobile mockup) up to desktop and tablet so the three form factors share a coherent visual language. Three changes drive the coherence: the primary controller pill arrives on desktop + tablet, the settings nav shows icons on every size, and tablet picks a side per orientation (portrait = mobile pattern, landscape = desktop pattern). The widget sheet vs right-pane split stays intentionally asymmetric — different real estate, different ergonomics.

What's the same as mobile

Primary controller pill
Horizontal floating pill anchored top-left of the map. Two states: Overview (no site loaded) → "All sites · N" label. Site loaded → "Overview" button + "›" separator + site chip; tapping Overview returns to the all-sites state. Right of the chip: zoom + home + bounds + basemap. Same component, same position across phone / tablet / desktop. Replaces ViewerSidebar's site-chip duty on desktop.
Site list strip
Shown only in the Overview state. Bottom of the map carries a horizontal scroll-snap strip of site cards (thumb + name + meta + Public badge). Replaces the widget rail one-for-one — widget rail is per-site, so it appears only after a site is loaded. Same component on every form factor; max-width 960px on desktop, full-width on phone.
Floating left icons
Vertical stack for the panel-bearing chrome — Search · Site info · Layers · Measure · Terrain · Legend. Acts as the primary sidebar (always visible). Map controls that already live in the pill (zoom / home / bounds / basemap) are not duplicated here.
Floating side panel
Pressing an icon slides out a 300 px panel docked to the right edge of the icon stack — the secondary sidebar. Single slot; other icons swap the panel content; second tap on the active icon (or ESC) dismisses. Same pattern VSCode uses for primary + secondary side bar. Measure is the exception — it's a tool toggle, no panel.
Mai FAB
Bottom-right, persistent across all sizes. defaultFabPos.y already calculated to clear stacked nav chrome.
Settings nav icons
Icon-left-of-label in the desktop rail, icon-above-label on tablet portrait + phone. Same icon set throughout. Drop the .shell.isPhone .navItemIcon gate.
Widget surface on desktop
Right pane (existing). Docked on the right edge, ~360px wide. Same widget components as the phone sheet, different render target.
Widget surface on tablet
Portrait → phone-style bottom widget sheet. Landscape → desktop-style right pane. Decided by orientation, not just breakpoint.
ViewerSidebar role
Shrinks. Site chip moves to the pill; LAYERS and MEASURE join the floating icon stack. Sidebar can disappear or collapse to a slim icon column.

Phased implementation plan

Phase 1 · Primary controller pill
Extract a shared CtrlPill component. Mount it on phone (already in mobile mockup), tablet portrait, tablet landscape, desktop. Behind it, kill the duplicate site-chip / map-controls instances in ViewerSidebar, CesiumViewer map-controls, etc.
Phase 2 · Settings icons everywhere
Drop the .shell.isPhone .navItemIcon gate. Render icons in desktop left rail too. Trivial CSS edit.
Phase 3 · Tablet orientation pivot
Introduce layoutMode derived from breakpoint × orientation. Replace tablet-drawer branch with portrait → phone-pattern, landscape → desktop-pattern.
Phase 4 · ViewerSidebar simplification
Once the pill owns site picker + basemap, ViewerSidebar shrinks. LAYERS / MEASURE entry points move to the floating icon stack (which already exists on phone). The sidebar can collapse entirely on desktop too, mirroring the mobile floating-icons pattern.
Phase 5 · Atlas Sites cards (optional)
Add a card carousel "map view" as a secondary tab on the desktop Atlas Sites page. Same cards as mobile, just wider with more visible at once. Defaults to the existing table view.
