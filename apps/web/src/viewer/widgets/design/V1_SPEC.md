## 1. SKETCH SYSTEM

### What a sketch is
A **sketch** is a user-owned, named container of CAD-like geometry nodes (the "design DAG") plus a schema, attached to one or more sites. It lives in three places at once:

1. **In-memory** — managed by the `useCadEngine()` composable as a DAG of nodes (`state.nodes`, `state.outputIds`).
2. **localStorage** — `mightydt_sketches` (the array of sketch records) + per-site recovery payload of all nodes. Crash-recovery only.
3. **S3 (per user)** — authoritative store. Two file kinds per user-per-site:
   - Index: `design-sketch-index-{siteId}.json` → manifest `{version: 2, siteId, activeSketchId, sketches:[{id,name,nodeCount,savedAt}]}`
   - Per-sketch: `design-sketch-{siteId}-{sketchId}.json` → `{version: 2, siteId, sketchId, sketch, nodes, savedAt}`

Save flow: state mutation → `markSketchDirty()` → 500ms debounced `saveDesignToS3()`; destructive ops (delete) call `flushSaveDesignToS3()` immediately. Schema-incompatible bumps live in `SKETCHES_SCHEMA_VERSION` (currently `4`).

### Sketch object shape (the in-memory + S3 record)
```js
{
  id: 'sketch_...',
  name: 'My drainage redline',
  siteIds: ['site_xyz'],            // affinity (single or multi)
  layers: [                         // user-visible draw layers (tier 1)
    {
      id, name, colour, visible, locked,
      coordMode, localOrigin, fields,    // optional per-layer overrides
      subSchemas: {                      // redline-only: per-geometry sublayer schemas
        point:  { fields: [...] },
        line:   { fields: [...] },
        polygon:{ fields: [...] }
      },
      presetValue,                       // tier-1 categorisation value (e.g. 'DRAINAGE')
      approval: { state, note, assignee } // UI-only flag, no backend enforcement
    }
  ],
  activeLayerId,
  coordMode: 'world' | 'local',
  coordCrs:  'EPSG:4326' | 'EPSG:7849-7856' | 'local',
  localOrigin: { lon, lat, alt },
  localRotation: 0,
  heightDatum: 'msl' | 'ahd' | 'ellipsoidal' | 'terrain',

  // Redline-only:
  targetDataSourceId,                    // single-layer redline target
  redline: {
    scope: 'layer' | 'site',
    targetLayerId,
    sublayerField,                       // e.g. 'asset_type' on the target table
    tables: { point: 'spatial_features_X', line: 'spatial_features_Y', polygon: '...' }
  },
  changeSet: {
    modified: [{ sourceDataSourceId, sourceFeatureId, sketchNodeId }],
    deleted:  [{ sourceDataSourceId, sourceFeatureId }]
  },

  // Sketch-level schema (default schema if layers don't override):
  fields: [{ id, key, type, defaultVal, auto? }],
  _schemaModified: false                 // dirty flag set by schema editor
}
```

### Node shape (what each feature looks like inside a sketch)
```js
{
  id: 'n...',                              // generated by useCadEngine
  type: 'sketch'|'model'|'extrude'|'pipe'|'box'|'pit'|'cylinder'|'string'|'loft'|...,
  inputs: ['n_upstream'],                  // DAG dependency
  template_id: 'at_...' | null,            // attribute-template id (style + default attrs)
  params: {
    geometry: 'point'|'line'|'polygon'|'other',
    positions: [[lon,lat,alt?], ...],      // WGS84 (anchor for solids = positions[0])
    sketchId,                              // owning sketch (lets engine route dirty)
    sketchLayer,                           // owning layer id (tier 1)
    // Solid params (box/pit/cylinder):
    width, depth, height, radius, heading, pitch, roll, refZ, wallThickness, floorThickness,
    // Pipe-specific:
    pipe_fields: { Size, Material, Configuration, ... }
  },
  attributes: { /* free-form key→value */ },
  style:      { color, opacity, lineWidth, pointSize, labelSize, ... }
}
```

### Lifecycle (create/edit/delete)
- **Create**: `createBlankSketch(name)` (factory) or `submitRedlineDraft(...)` (redline fork).
- **Edit attributes/positions/style/params**: `updateNodeAttributes`, `updateNodePositions`, `updateNodeStyle`, `updateNodeParam` from `useCadEngine`. Each call → `markDirty(nodeId)` + downstream → `evaluate()` (or deferred if `liveHistoryEnabled=false`) + `markSketchDirty(sketchId)` + debounced S3.
- **Delete**: `removeNode(id)` cascades to dependants (e.g., delete a polyline → its pipe goes too); persists to S3.
- **Undo/redo**: `useCadEngine().undo()/redo()` — max 100 steps. Cesium primitives (`_meshPrimitives`) and DataSource entities are reconciled on undo.

### Relationship to sites/layers/features
- **Site** → many sketches (siteIds affinity). `visibleSketches` filters the gallery to current site, but ALL sketches still render on the globe.
- **Sketch layers ≠ site layers.** Sketch layers are draft containers tied to colour/preset/schema. A redline sketch's layers map onto silent geometry sublayers backed by PostGIS tables; a freeform sketch's layers are purely user-defined.
- **Feature**: a `node` is the unit. Features in a redline can reference (and modify/delete) existing rows of the target PostGIS table via `changeSet`.

## 2. REDLINE SKETCH SYSTEM

### What makes a sketch a "redline"
A redline sketch carries a `targetDataSourceId` and a `redline: { scope, targetLayerId, sublayerField, tables }` block. The sketch is **site-locked** to the target's site. Two scopes:

- **scope='layer'**: target a single PostGIS table set (one data source). Schema + CRS inherited from that data source.
- **scope='site'**: target every layer in a site. Built by `buildWholeSiteRedline()` which aggregates each site layer into a sketch layer (tier 1) and resolves three tables per layer (point/line/polygon).

### Two-tier architecture
- **Tier 1** = user-facing sketch layer with a `presetValue` (categorisation). Data-driven from target's sublayer-categorisation preset.
- **Tier 2** = silent geometry sublayers under each tier-1 layer, mapped to per-geometry PostGIS tables (`tables.point`, `tables.line`, `tables.polygon`).

### Schema inheritance
- On redline creation, `importSchemaFromTargetLayer()` calls `GET /api/data-sources/{id}/fields` and populates `layer.fields` (shared) + `layer.subSchemas.{point,line,polygon}.fields` (per geometry).
- Constraints: the legend → `sublayer_field` enum is only correct under universal autodetect; backend `data_sources.source_srid` MUST be populated for CRS to inherit.

### CRS inheritance ("4-step fallback")
Frontend probes `GET /api/data-sources/{id}/source-crs` for every data source on the site (lazy, populates `sourceCrsByDsId`). Backend tries in order:
1. `data_sources.source_srid` column (most authoritative)
2. `data_source_crs.crs_code` table join
3. `ST_SRID(geom_src)` from the `*_vw_4326` view (real PostGIS metadata, ≥1 row)
4. JSON in `source_crs_info` or `metadata` column → keys `source_epsg`/`source_srid`/`epsg`

Response includes `{epsg, source: 'data_sources_column'|'data_source_crs_table'|'geom_src_probe'|'metadata_json'|'unknown'}` so the UI can degrade — `unknown` falls back to 4326 with a banner.

The widget computes `redlineValidation` with three inheritance states: `pending` | `complete` | `failed` (CRS mismatch across site layers blocks scope='site' and demotes to scope='layer').

### "Edit string in redline" flow (special)
A "string" is a polyline aggregate from IFC import where each vertex was also imported as a child point feature. Endpoint `GET /api/data-sources/{id}/string-group?polyline_feature_id=...` returns the polyline + every child point ordered by `_vertexIndex`. Match logic:
- Primary: child points where `properties->>'_parentPolylineId' = polyline.feature_id`
- Fallback: where `properties->>'_aggregateRelId' = polyline._aggregateRelId`
- Casing-aware (also checks `parentPolylineId` without underscore)

The widget picks up the bundle as one editable unit.

### Workflow: create → submit → review → promote

```
LayersTab "+ New" → Redline tile
  → Modal: site → scope (layer|site) → target → name
  → submitRedlineDraft() → frontend creates sketch w/ redline metadata
  → Schema + CRS auto-imported (async, banner shows pending/complete/failed)

User draws features (SketchTab tools) → tracked in nodes
User edits existing features → entry added to changeSet.modified
User deletes existing features → entry added to changeSet.deleted
Orphan nodes (geom kind has no target table) → flagged with ⚠ in tree

DownloadTab → Submit for Review:
  POST /api/design/submit
    body: { site_id, sketch_name, sketch_data: <full sketch>, node_count, notes, schema_changes? }
    → submission row inserted (status='pending'); user sees in "My Submissions"

Admin (web admin app):
  GET /api/admin/submissions[?status&site_id]      → queue
  GET /api/admin/submissions/{id}                  → detail incl. full sketch_data
  POST /api/admin/submissions/{id}/plan            → preview promotion plan (no DDL/DML)
  PATCH /api/admin/submissions/{id}/approve        → status='approved'
  PATCH /api/admin/submissions/{id}/reject         → status='rejected'

If schema_changes pending (Phase 4 — pipe submissions w/ unmapped canonical fields):
  POST /api/admin/submissions/{id}/approve-schema-changes
    → validate each entry → ALTER TABLE ADD COLUMN IF NOT EXISTS in single TX
    → log every DDL into schema_change_log
    → mark schema_changes_approved_at=NOW()
    Allowed actions: 'add_column' only.
    Allowed types: TEXT|INTEGER|REAL|DOUBLE PRECISION|NUMERIC|BOOLEAN|TIMESTAMPTZ|JSONB
    Allowed table prefixes: 'spatial_features_' or 'site_'

Promotion (gated on status='approved' AND schema_changes_approved_at if present):
  POST /api/admin/submissions/{id}/promote
    → _build_promotion_plan(sketch_data) → plan {inserts, updates, deletes, orphans, summary}
    → _inject_wkt_into_plan(plan, sketch_data) → attach WKT to each entry
    → _execute_plan_pg(conn, plan) → per-entry try/except, partial success allowed
    → status='promoted', promoted_at=NOW()
```

### Promotion plan internals (`_build_promotion_plan`)
- Identify polylines consumed by pipes → mark them as orphans (pipe carries the geometry).
- Each non-modified node → INSERT entry: route via `redline.tables[geomKind]`, build attrs from layer schema (subSchemas[geomKind] → fallback to layer.fields), inject `presetValue` into `sublayerField` column, fuzzy-match canonical pipe fields → target columns.
- Each `changeSet.modified` node → UPDATE entry (carries source_data_source_id, sourceFeatureId).
- Each `changeSet.deleted` → DELETE entry (table left null; admin must resolve manually — known gap).
- Orphans logged with reason ("geometry kind unsupported", "no table", "consumed by pipe").

### Promotion execution (`_execute_plan_pg`)
- Verify each target table in `information_schema.tables` (schema='public').
- Detect columns from `information_schema.columns`; drop unknown attrs silently.
- Detect geometry column from `geometry_columns`; fall back to 'geom'/'geometry'/'the_geom'/'shape'.
- INSERT uses `ST_GeomFromText(wkt, 4326)`.
- UPDATE uses id_col ('id' or 'objectid') WHERE clause.
- Per-entry rollback on error; record `err_inserts/err_updates/err_deletes` and continue.
- SQLite path is a no-op (dev only).

## 5. PANEL STRUCTURE

The widget has **one rail + one panel**. The rail switches `designMode`. The panel renders one of six tabs (extracted SFCs) and may overlay the **place-mode bar** (when an active draw tool is set) or the **modify-props bar** (when a node is selected and no tool active).

### Rail / `designMode`
| Mode | Tab SFC | Purpose |
|---|---|---|
| `layers` | `LayersTab` | Sketch gallery, redline modal, schema editor, sketch settings, preset selector, layer list |
| `sketch` | `SketchTab` (+ place-mode-bar overlay) | Tools grid (Create/Modify), template browser, op param bar |
| `features` | `FeaturesTab` | Tree + table view of all features; drag-drop, batch select, attribute table w/ CSV import diff |
| `properties` | `PropertiesTab` | Selected node detail (Move/Rotate/Scale + DesignObjectEditor) |
| `download` | `DownloadTab` | Import objects (GLB/IFC), import geometry, export, **Submit for Review** |
| `history` | `HistoryTab` | DAG history, By-Layer / By-Type, live-history toggle + Rebuild |

### Place-mode bar (visible when `activeTool && designMode==='sketch'`)

### What each tool writes to the data model
- Most tools: a sketch node `{ type:'sketch', params:{geometry, positions, sketchLayer, sketchId, ...toolParams}, attributes, style, template_id }`.

- Smooth look-at on zoom (HARD CONSTRAINT): on first wheel after object selection, ease to selected — don't snap. This is MightyDT's strength vs Google Earth.

## 7. EXPORT SYSTEM

### Frontend
DownloadTab → Export Geometry section. User picks Format / CRS / Sketch scope (all or one) / Split mode (none / by layer / by attribute → attribute name field). Calls `POST /api/design/export` with the resolved feature list.

### Backend (`app/modules/design_export/routes.py`)
Supported formats:
| Format | Library | Notes |
|---|---|---|
| GeoJSON | builtin json | indent=2, `application/geo+json` |
| CSV | builtin csv | `geometry_wkt` column + property keys; via Shapely WKT |
| Shapefile (zip) | geopandas + fiona | Grouped by geometry type (Shapefile = uniform); zipped |
| KML | geopandas (KML driver) | `application/vnd.google-earth.kml+xml` |
| GeoPackage | geopandas (GPKG driver) | `application/geopackage+sqlite3` |
| DXF | ezdxf (R2010) | Layer = properties.layer (default '0'); Point→add_point; LineString→add_lwpolyline; Polygon→add_lwpolyline(close=True) |

CRS preset list (`GET /api/design/export/crs-options`): WGS84, GDA94, GDA2020, all MGA zones (GDA94+GDA2020), NZTM (2193), UTM 56S (32756). Reprojection via `pyproj.Transformer.from_crs(..., always_xy=True)` recursively over coordinates.

Streaming response with `Content-Disposition: attachment; filename="{filename}.{ext}"`.

### Import (parallel module)
`POST /api/design/import` accepts geojson/json, shp/zip, gpkg, kml/kmz, dxf, csv/tsv. Returns `{filename, extension, crs_detected, crs_epsg, feature_count, geometry_counts, field_schema, features}`. CSV auto-detects lon/lat/alt column aliases; DXF parses POINT/LINE/LWPOLYLINE/POLYLINE/CIRCLE(64-pt poly)/ARC(5° interp)/SPLINE(flattened); preserves `properties.layer`. CRS detection delegated to `app.modules.data_processing.crs_detection.detect_crs_from_file()`.

## 8. ATTRIBUTE TEMPLATES

### What they are
Reusable bundles of `{ attributes:{...}, fields:[...], style?:{...}, geometry:'point'|'polyline'|'polygon' }`. Applied either via the SketchTab template grid (changes active template + auto-activates the matching tool) or via the `<AttributesEditor>` "save as template" button (snapshot current values → register as new template).

### Where stored
Three layers — merged at runtime:

1. **Site-level (Postgres-backed)** — `sites.metadata.design_templates` array. CRUD via:
   - `GET/PUT /api/sites/{site_id}/design-templates` (PUT requires creator/admin/super_admin + site access; max 500 templates, ≤900KB JSON).
2. **User-level per site (S3-backed)** — `/api/me/json-files/design-templates-{sanitized-siteId}.json` (raw array).
3. **Auto-synthesised from definition key** — `autoDefinitionKeyTemplates` (computed) crosses every site layer × every definition_categories.codes entry to create on-the-fly templates with the legend colour. Not persisted; computed each render.

Looked up via `findAssetByTemplateId(templateId)` — checks `supplementalTemplatesById` (definition-key auto + custom merged) first, then `state.assetCatalog`.

### How applied
- User picks template → `state.activeTemplateId` set → AttributesEditor renders fields with `defaultValue`s.
- Tool placement → new node created with `template_id` set + `attributes` populated from template.
- Template "fields" drives the rows shown in AttributesEditor; "style" merges over `DEFAULT_NODE_STYLE`.

### Canonical pipe template (special — `src/utils/pipes3DConfig.js`)
Pipe attributes use a fixed canonical schema (Size, Configuration, Material, AssetType, DepthReference, WallThickness, Name, Status, Owner, InstallDate) with aliases. Both layer autodetect and redline column resolution use the same fuzzy matcher (`resolveCanonicalToTarget`) so the same field names flow both ways. UOM list: `mm/cm/m/in/ft/km`. Depth references: `outsideTop|obvert|centerline|invert|outsideBottom`.

Default canonical pipe fields:
```js
[
  { key: 'Size',           role: 'diameter',      uom: 'mm', aliases: ['NominalDiameter','Diameter','DN','Dia'], defaultValue: 100 },
  { key: 'Configuration',  role: 'bankConfig',               aliases: ['Config','BankConfig','PipeBank'], defaultValue: '' },
  { key: 'Material',       role: 'material',                 aliases: ['Mat','PipeMaterial'], defaultValue: '' },
  { key: 'AssetType',      role: 'assetType',                aliases: ['Type','Subtype','Class'], defaultValue: '' },
  { key: 'DepthReference', role: 'depthRef',                 aliases: ['DepthRef','PlacementRef','Reference'], defaultValue: 'outsideTop' },
  { key: 'WallThickness',  role: 'wallThickness', uom: 'mm', aliases: ['WT','WallThk'], defaultValue: 0 },
  { key: 'Name',           role: 'label',                    aliases: ['Label','PipeName'], defaultValue: '' },
  { key: 'Status',         role: 'status',                   aliases: ['State','Condition'], defaultValue: '' },
  { key: 'Owner',          role: 'owner',                    aliases: ['Operator','Custodian'], defaultValue: '' },
  { key: 'InstallDate',    role: 'installDate',              aliases: ['Installed','CommissionDate'], defaultValue: '' },
]
```

## 9. CRITICAL ARCHITECTURAL NOTES FOR THE V2 PORT

These are the patterns/gotchas to flag for a faithful React + FastAPI port:

1. **Provide/inject is the spine.** The shell exposes a single `designCtx` reactive bundle of refs+functions; tabs are pure presentation. Equivalent in React: a Context (or Zustand store) co-owned by the shell, with each tab consuming via a typed hook. Don't prop-drill.

2. **DAG engine is portable as-is.** `useCadEngine.js` is logic-heavy but framework-light — porting to a Zustand store is mostly mechanical. Preserve: dirty-set propagation, topological sort, per-sketch dirty tracking (S3 sync gating), undo/redo with cascade-delete + Cesium primitive cleanup, `_persistReady` guard against overwriting recovery before mount restore.

3. **Tool-Registry "Object Sheet" pattern.** Adding a tool = drop a record + create one Parameters SFC. Keep that pattern in v2; the alternative (nested if/else in a 1000-line render) is unmaintainable.

4. **Three-tier sketch storage.** localStorage (recovery) → S3 per-user-per-site (authoritative) → Postgres (only for shared site templates/presets). Don't collapse to one. Sketches themselves are user-private — they don't go in Postgres until *promoted*.

5. **4-step CRS fallback is load-bearing.** Always include the `source` discriminator in the response so the UI can degrade. Frontend must treat `'unknown'` as "fall back to 4326 + warn".

6. **Per-table dynamic spatial schema.** `spatial_features_{data_source_id}` + `_vw_4326` view pattern survives in v2. Geometry stored in source SRID; view exposes both `geom_src` (original) and `geometry` (WGS84). All design widget queries should hit the view unless they need the original SRID.

7. **Promotion plan is the contract.** The plan structure (inserts/updates/deletes/orphans/summary) is consumed by the admin diff UI before promote. Keep the JSON shape stable. The admin web app needs a "preview before promote" workflow.

8. **Schema-changes are gated.** Promote MUST refuse if `schema_changes` non-empty AND `schema_changes_approved_at` null. Allowed actions = `add_column` only; allowed table prefixes = `spatial_features_` or `site_`; allowed types = the 8 in the validator. Per-DDL audit row in `schema_change_log`.

9. **Naming convention:** for parallel variants in code, use descriptive names (classic/new), not v1/v2. Already too many splits on that nomenclature.

10. **Smooth look-at on zoom:** MightyDT's strength vs Google Earth — first wheel after selection eases to the selected object. Don't lose this in v2.

11. **Site-lock for redlines:** A validated redline is single-site (one set of PostGIS tables). The widget should refuse multi-site affinity for redlines.

12. **Pipe canonical fields are shared (`pipes3DConfig.js`).** Front and backend duplicates. In v2, host this once and let both sides import.

13. **Draw.vue is dead code in the new path.** `DesignWidget.vue` inlines its own draw logic; only the archived `DesignWidget.archive-2026-03-28.vue` imports `<Draw />`. Don't port Draw.vue — it's superseded by the place-mode-bar + tool registry.

14. **Sketch ID regeneration during preset restore** — `reconcileNodeSketchLayers()` remaps `node.params.sketchLayer` to valid layer IDs after a preset apply regenerates layer IDs. v2 must keep this fix-up step.

15. **Known gaps / dragons** worth carrying as TODOs into v2:
   - Delete records in `changeSet.deleted` don't carry geometry kind → `_build_promotion_plan` leaves `table=null` for deletes; admin must resolve manually.
   - design_models has `thumbnail_s3_key` column but no thumbnail generator implemented.
   - No optimistic locking on submissions; concurrent edits = last write wins.
   - SQLite path in promotion is a no-op — fine for dev, but means PG-only for any integration test that hits promote.
